# Independent Code Review

Date: 2026-05-20

Scope: current working diff in the Hong Kong quarterly macro-econometric project. This review follows `CLAUDE.md` as active project memory and focuses on real defects, stale research outputs, data integrity, and verification gaps.

## Proposed Untangling Plan

The core problem is not just individual bugs. The project currently mixes four different things in the same scripts and output folder:

1. data refresh,
2. model estimation,
3. validation/audit checks,
4. research-note writing.

That makes the project feel more like an engineering pipeline than a compact macroeconometric research note. It also makes stale outputs easy to quote by accident because "latest output" depends on whichever command ran last.

### Separate research artifacts from machinery

Use three layers:

- `data/`: source and canonical panels only.
- `results/`: small, stable model result tables for the paper.
- `output/`: disposable plots/logs generated by the latest run.

The paper and README should quote only from `results/`, not directly from `output/`. The `output/` folder can still exist for diagnostics, but it should not be treated as the source of truth for final claims.

Suggested stable result files:

- `results/property_vecm_headline.csv`
- `results/seven_variable_var_benchmark.csv`
- `results/forecast_comparison.csv`
- `results/model_limitations.md`

Each result file should have a short header or sidecar metadata saying which command generated it, when, and from which data file.

### Make the command structure boring

Instead of one script doing everything, split commands by research question:

```bash
python fetch_real_data.py
python hk_var_model.py --include-property --model-type vecm --lag-criterion bic --save-tag property_vecm
python hk_var_model.py --model-type var --lag-criterion bic --save-tag var_benchmark
python build_research_tables.py
```

The first two commands estimate models. The final command should assemble clean paper-ready tables from saved result files. That prevents the paper from accidentally mixing a current VECM chart with stale VAR text.

### Keep one headline model and one benchmark

The hierarchy should be explicit everywhere:

- Main model: property-channel VECM.
- Benchmark: seven-variable VAR.
- Robustness: BVAR only if actually run and reported.

Do not let the code, paper, and README describe different model hierarchies. If a table is benchmark-only, label it benchmark-only. If a chart is model-space rather than level-space, label it model-space.

### Move engineering checks out of the reader path

The README should not read like a software release note. Keep it to:

1. research question,
2. data,
3. model hierarchy,
4. main result,
5. limitations,
6. exact reproduction commands.

Put engineering details in separate files:

- `VALIDATION.md` for tests, ruff, and diagnostics.
- `DATA_PROVENANCE.md` for source details.
- `code_review.md` for defects and fixes.

The paper should not mention test suites, file-generation mechanics, or audit history unless directly relevant to model credibility.

### Make the research note more accessible

Open with the economic story before the econometrics:

> Hong Kong imports US interest-rate conditions through the currency board, while China demand affects Hong Kong through trade, prices, labor-market conditions, and asset prices. The model asks which channels account for the largest forecast-error shares after shocks to US monetary conditions and China growth.

Then use one short table:

| Channel | Main result | Interpretation |
| --- | --- | --- |
| US FFR to HIBOR | large FEVD share | currency-board interest-rate transmission |
| US FFR to unemployment | material FEVD share | tighter monetary conditions show up in labor-market slack |
| China growth to GDP/CPI | material but model-dependent | China activity matters, but precision is limited |
| Property prices | asset-price channel | relevant, but less precisely estimated |

Keep the language reduced-form:

- Use "transmission evidence", "dynamic association", and "forecast-error share".
- Avoid "causal effect", "identified shock", and "structural estimate" unless the identification strategy is strengthened.
- Use "US monetary conditions" for `us_ffr`, not "pure monetary policy shock".

### Reduce the paper to one clean spine

A compact final note can be organized as:

1. **Question**: US monetary and China growth transmission to Hong Kong under the currency board.
2. **Data**: quarterly official data, 1998Q1-2026Q1, plus property-price channel.
3. **Model**: property-channel VECM headline, seven-variable VAR benchmark.
4. **Results**: FEVD at 8 quarters, IRF direction, forecast RMSE.
5. **Limitations**: residual autocorrelation, VECM rank sensitivity, `us_ffr` proxy limitation, property-price precision risk.
6. **Next extension**: shorter monetary-surprise sample or property-growth robustness.

This makes the project readable as macroeconomics first, with code only supporting the claims.

## High Severity

### VECM rolling backtest compares level forecasts to transformed actuals

Location: `hk_var_model.py`, `rolling_backtest()` and main call around `STEP 4`.

Trigger: run the property-channel VECM path:

```bash
MPLCONFIGDIR=/tmp/mpl_cfg python hk_var_model.py --include-property --model-type vecm --lag-criterion bic
```

In the VECM branch, forecasts are generated from `df_raw` levels, while `actual_block` comes from `df_est`, where several variables are first-differenced. Confirmed example: `china_gdp` transformed actual was `-0.3` while raw actual was `7.6`; `hk_property_price_idx` transformed actual was `3.7` while raw actual was `241.1667`.

Impact: `05_backtest_h1.png`, `05_backtest_h4.png`, and the generic VECM `bt_summary` report numerically invalid RMSEs.

Fix: for `model_type == "vecm"`, compute actuals and AR/RW benchmarks from `df_raw` in level space, or skip this generic backtest for VECM and use only `compare_var_vecm_backtest_level()` for VECM reporting.

### Historical decomposition mixes model-space and level-space data

Location: `hk_var_model.py`, main call to `compute_historical_decomposition(result, df_est, var_names)` and `plot_historical_decomposition(..., df_raw=df_raw)`.

Trigger: any VECM run, and also transformed VAR/BVAR runs. VECM coefficients and residuals are level-space, but decomposition history is passed as `df_est`. The plot then overlays raw-level actuals.

Impact: `08_hist_decomp.png` is not a valid decomposition of the plotted series. Bars and actual line can be in different units.

Fix: for VECM, call decomposition with `df_raw`. For transformed VAR/BVAR, either plot actuals from `df_est` and label the chart as model-space, or reconstruct level-space contributions before overlaying raw actuals.

### Project memory can disappear on checkout

Location: `AGENTS.md` points to `CLAUDE.md`, while `.gitignore` ignores `CLAUDE.md`.

Trigger: fresh checkout or another machine where `CLAUDE.md` is not locally present. `git ls-files` shows `CLAUDE.md` is not tracked, and `git check-ignore -v CLAUDE.md` hits `.gitignore`.

Impact: agents receive an `AGENTS.md` pointer to a missing ignored file, so the core project constraints disappear.

Fix: remove `CLAUDE.md` from `.gitignore` and commit it, or move the required project memory into a tracked file.

### Paper is stale against current generated outputs

Location: `paper/main.tex`.

Trigger: the paper still describes a seven-variable VAR as baseline and a rank-2 non-property VECM as robustness. Current project memory and outputs use the property-channel VECM as the headline model, with Johansen rank 1 and different h=8 FEVD values.

Impact: the main research note reports stale model hierarchy and stale numbers. One direct mismatch: the paper says the VECM wins 18 of 28 RMSE cells, while `output/var_vecm_backtest_comparison.csv` has 24 cells split 12 VAR / 12 VECM.

Fix: rewrite the paper around the property-channel VECM as the main model, rank 1, and current FEVD table. Keep seven-variable VAR numbers only in a clearly labeled benchmark table generated from a separate benchmark artifact.

## Medium Severity

### CPI fallback does not run when official CPI fetch fails

Location: `fetch_real_data.py`, `fetch_hk_cpi()`.

Trigger: `fetch_hk_cpi_official()` raises a network, API, JSON, or parse exception.

Impact: the documented World Bank fallback only runs when the official result is empty; an official-source outage aborts the data refresh.

Fix: wrap the official CPI fetch in expected fetch/parse exception handling, then call `fetch_hk_cpi_wb_fallback()` and record fallback lineage/source.

### China GDP fallback can silently truncate and mislabel the canonical sample

Location: `fetch_real_data.py`, `fetch_china_real_gdp_oecd()` and `build_dataset()`.

Trigger: OECD QNA fetch/parsing fails, so the code falls back to FRED nominal GDP. The mean-based source label can still print `OECD_QNA_B1GQ_GY`, and the fallback series may end earlier than the canonical 1998Q1-2026Q1 sample.

Impact: `merged.dropna()` can silently shorten the canonical panel while provenance still implies real OECD data.

Fix: return `(df, source_id)` from the China fetcher, write the actual source to metadata, and add a post-merge coverage assertion for expected first/last quarter and row count.

### VECM lag mapping contradicts BIC lag selection

Location: `hk_var_model.py`, VECM lag mapping.

Trigger: BIC selects `p=1`, which is the current property path. The code maps this to `vecm_lag_diff = max(lags - 1, 1)`, forcing `k_ar_diff=1`, meaning two level lags.

Impact: diagnostics report selected lag 1, while the headline VECM is estimated with two level lags. This affects rank testing, FEVD, IRF, forecasts, and backtests.

Fix: allow `k_ar_diff=0` when the selected VAR lag is one. Keep validation only against negative explicit CLI values.

### Methods note claims crisis dummies are included for VECM, but VECM ignores them

Location: `hk_var_model.py`, crisis dummy construction and `write_methods_note()`.

Trigger: `model_used == "vecm"`. Dummies are passed into `estimate_model()`, but the VECM branch does not pass exogenous variables into `statsmodels.VECM`.

Impact: the methods note overstates break treatment and can hide residual autocorrelation risk that should remain visible.

Fix: either pass aligned dummies through `fit_vecm(..., exog=...)`, or condition the methods note so the dummy claim appears only for VAR/BVAR fits where exog is actually used.

### Scenario forecast dates start on the last observed quarter

Location: `hk_var_model.py`, forecast date construction.

Trigger: canonical data end at `2026-01-01`, but scenario forecasts also start at `2026-01-01`. The first forecast values are not the observed data.

Impact: forecast tables and plots can be read as replacing the observed 2026Q1 quarter with a forecast.

Fix: use a quarter-start offset aligned to the panel, such as `pd.offsets.QuarterBegin(startingMonth=1)`, so the first out-of-sample forecast after `2026-01-01` is `2026-04-01`.

### Scenario shock tests no longer cover documented level-space semantics

Location: `tests/test_hk_var_model.py`.

Trigger: `forecast_scenarios()` no longer uses `_shock_in_level_space()` and directly adds first-step model-space deltas. The current integration test constructs every transform as `"level"`.

Impact: tests can pass while differenced-variable scenario semantics regress.

Fix: add an integration test using `sample_transforms`, especially `us_ffr` as `first_diff`, and assert the intended level-space scenario path. If the new intended behavior is first-step model-space innovations, rename/remove stale level-space helper tests and update labels.

### Cursor auto-run rule is weaker than project verification

Location: `.cursor/rules/hk-var-auto-run.mdc`.

Trigger: Cursor auto-rule runs after `tests/` or `hk_var_model.py` edits.

Impact: code can be declared usable with passing tests while lint fails or the headline property VECM path is broken. The rule also uses "propagate", conflicting with the project wording rule to use "transmit/transmission".

Fix: add `python -m ruff check .`, add the property-channel VECM command for model edits, and replace "propagate" with "transmit".

## Low Severity

### Stale China real GDP sidecar can mislead provenance checks

Location: `data/china_real_gdp_quarterly.csv`.

Trigger: a reader or downstream script uses this sidecar as the China real GDP source.

Impact: the sidecar can drift from the canonical panel because `fetch_real_data.py` does not regenerate it.

Fix: remove it from the changed set, or make the OECD fetcher write/update it with the same range and documented column mapping.

### Synthetic-data fixture seed is ineffective

Location: `tests/test_hk_var_model.py`, `synthetic_data` fixture.

Trigger: test order changes or repeated fixture calls. The fixture calls `np.random.seed(42)`, but `_generate_calibrated_data()` uses module-level `np.random.default_rng`.

Impact: tests can become order-dependent.

Fix: reset or monkeypatch `m._RNG_DATA = np.random.default_rng(m.SEED)` in the fixture, or refactor `_generate_calibrated_data()` to accept an RNG.
